# MISSION: Expand vCenter Alarm Monitoring (Folder/License Alarms)

**Date:** 2026-03-26 16:25 UTC  
**Source Branch:** feature/vc-folder-alarms-fix  
**Change Type:** minor  
**Previous Version:** v1.10.0

## Objective / Mission
The primary objective was to resolve visibility gaps in the vCenter (`VC`) module:
1.  **Alarm Scope**: Certain triggered alarms, specifically those associated with **Folder** or **Datacenter** objects (e.g. licensing), were missing from DCI reports.
2.  **Infrastructure Transparency**: Hosts with connectivity issues (Disconnected/Not Responding) or in unexpected power states (Powered Off/Standby) were not being detailed in the DCI report, causing a mismatch between the Summary "Critical" status and the Investigative details.

## Analysis / Background
In vSphere environments, alarms are triggered on specific Managed Entities. However, these alarms often "propagate" to parent containers (Folders, Clusters, Datacenters). 
1.  **Scope Limitation**: The previous implementation only looked for alarms on Hosts, VMs, Datastores, Clusters, and Networks. Top-level folders or Datacenter-level alarms (like licensing) were skipped.
2.  **Propagation Noise**: Simply adding all object types to the collection view would result in duplicate entries (e.g., a Host alarm appearing multiple times on its parent folders).
3.  **Missing Context**: Users requested better visibility into who acknowledged an alarm and when, to better match the vCenter UI experience.

## Implementation Details
- **Improved Host Visibility**: Implemented a new "Host Connectivity Detail" table in the DCI report. This table automatically filters to show only the ESXi hosts that require attention (all those not in `CONNECTED` and `POWERED_ON` state).
The solution implements a refined collection strategy:
- **Broadened Collection View**: Updated the SOAP `CreateContainerView` to include `vim.Folder`, `vim.Datacenter`, and `vim.ResourcePool`.
- **Source-Only Filtering**: Added a logic check (`alarm_state.entity == entity`) to ensure an alarm is only recorded for the object on which it was **directly triggered**. This prevents duplicates from parent folders while ensuring top-level alarms are caught.
- **Enhanced Data Structure**: Added `acknowledged_time` and `acknowledged_by` to the data collection and processing pipeline.
- **DCI Reporting**: The `alertsDetail` CSV now includes these additional columns for better troubleshooting context in the DCI Investigative report.
