# New Module: ESX Standalone Monitor

## Background

The existing `VC` module requires a vCenter Server. In environments where an ESXi host runs standalone (without vCenter), the vSphere REST API is unavailable. All data must be collected using the **vSphere Web Services (SOAP) API** via **pyVmomi**, which works directly on the ESXi host.

The new module will be called **`ESX`** and will follow the exact same 4-script structure as existing modules (VC, DD, PPCR, etc.).

> [!IMPORTANT]
> The `pyVmomi` library is already a dependency of the project (used in `VCgetinfo.py`). No new Python packages are required.

## User Review Required

> [!IMPORTANT]
> **Module Name**: I propose calling it `ESX` (folder `src/ESX/`). The config section key will also be `ESX`. Please confirm or suggest an alternative name.

> [!NOTE]
> **Data Scope**: An ESXi standalone host exposes less information than vCenter. The proposed scope covers:
> - **Host hardware/health status** (connection state, power state, CPU/RAM summary)
> - **Datastores** capacity and free space
> - **VMs** and their power state
> - **Triggered alarms** (via SOAP, same logic as VC)
>
> There is **no appliance health REST endpoint** on standalone ESXi. Host health will be derived from the SOAP `HostSystem` object's `overallStatus` field.

### New Module — `src/ESX/`

#### [NEW] `__init__.py`
Empty file to make ESX a Python package (same as VC).

---

#### [NEW] `ESXgetinfo.py`

All data collection via **pyVmomi SOAP API only** (no REST). Reuses the SOAP connection pattern from `VCgetinfo.py`.

**Data collected per instance:**

| JSON file | Source | Description |
|---|---|---|
| `host_health.json` | `HostSystem.overallStatus` | Overall health (`green`, `yellow`, `red`) |
| `datastores.json` | `HostSystem.datastore` | Capacity, free space, name, type |
| `vms.json` | `HostSystem.vm` | VM name, power state |
| `alarms.json` | `triggeredAlarmState` | Triggered alarms — reuses VC's existing `fetch_soap_alarms()` logic by importing it |

> [!TIP]
> `ESXgetinfo.py` will **import and reuse `fetch_soap_alarms`** directly from `VCgetinfo.py` to avoid code duplication. Only a single SOAP connection is needed per instance.

**Main flow `getinfo(dcocfg, **kwargs)`:**
1. For each instance in `dcocfg.instances("ESX")`:
   - Read credentials via `dcocfg.loginInfo("ESX", instance)`.
   - Open one SOAP connection (`SmartConnect`).
   - Collect host health, datastores, VMs, and alarms.
   - Save to JSON via `dcocfg.save_json(...)`.
   - Disconnect.

#### [NEW] `ESXprocessinfo.py`

**Summary CSVs generated:**

| CSV file | Purpose |
|---|---|
| `systemSummary.csv` | DC report — rows: Host Health, Datastore Capacity, Active Alerts, VMs Powered Off |
| `datastoreStatus.csv` | DCI report — capacity detail per datastore |
| `vmStatus.csv` | DCI report — VMs not POWERED\_ON |
| `alertsDetail.csv` | DCI report — triggered alarms (same columns as VC) |

Processing logic mirrors `VCprocessinfo.py`:
- Host health derived from `overallStatus` → mapped to `OK / Warning / Critical`.
- Datastore status uses same thresholds as VC (Green <85%, Yellow 85-95%, Red ≥95%).
- Alarms: only unacknowledged alarms affect the aggregate status.

**Entry point:** `proccess_info(dcocfg, **kwargs)` (same spelling convention as all other modules).

---

#### [NEW] `ESXcreatereportDC.py`

Summary table added to report section `"Compute"` under title `"ESX"`.

Rows displayed (same style as VC):
- Host Health
- Datastore Capacity
- Active Alerts

VMs Powered Off **not** shown in summary (same as VC design decision).

**Entry point:** `create_DC(dcocfg, dcorpt)`

---

#### [NEW] `ESXcreatereportDCI.py`

Detailed tables, in order:
1. **Datastore Capacity Detail** — all datastores, color-coded by % Used.
2. **VM Power Status Detail** — only VMs not `POWERED_ON`, highlighted in red.
3. **Alerts Detail** — same columns and coloring as VC module.

**Entry point:** `create_DCI(dcocfg, dcorpt)`
