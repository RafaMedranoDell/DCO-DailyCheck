# Bugfix Release — v1.15.2

**Date:** 2026-04-30  
**Branch:** feature/v1.15.2-fix-ppcr-nan-policies  
**Version:** v1.15.2  

## Problem

During the `process` phase of the PPCR module, the script crashed with the following error when processing systems where one or more **policies had no modification date** (`modifiedDate = null` in the API response):

```
ValueError: cannot convert float NaN to integer
```

This typically occurs with policies that have been **recently created** or have **never completed a protection cycle**, causing PPCR to return a `null` timestamp for `modifiedDate`.

## Root Cause

Two points in the code lacked protection against `NaN` or empty values:

1. **`PPCR/PPCRprocessinfo.py` (line 61)**: The elapsed time calculation resulted in `NaN` when `modifiedDate` was `null`, causing a crash later in `format_duration()`.

2. **`common/functions.py` — `df_timestamps_to_dates()`**: The function used `int(x)` inside a map, which failed with `ValueError: invalid literal for int() with base 10: ''` when encountering empty strings from the PPCR API.

## Fix Applied

### `PPCR/PPCRprocessinfo.py`
Added `.fillna(9999999)` after computing `elapsed_seconds`, before calling `format_duration`. Policies without a `modifiedDate` are now treated as "very old" (sentinel value), ensuring they are excluded from the OK count and display `"N/A"`.

### `common/functions.py` — `format_duration()` & `df_timestamps_to_dates()`
- **`format_duration()`**: Now returns `"N/A"` for `None`, `NaN`, or infinite values.
- **`df_timestamps_to_dates()`**: Added a `safe_convert` wrapper that handles `None`, empty strings `''`, and `NaN`. It now returns `pd.NaT` (Not a Time) instead of crashing.

## Impact

| Module | Impact |
|---|---|
| `PPCR/PPCRprocessinfo.py` | ✅ Fixed: no longer crashes on null/empty policy dates |
| `common/functions.py` | ✅ Hardened: `format_duration` and `df_timestamps_to_dates` now safe |
| `DD/DDprocessinfo.py` | ✅ Protected preventively: uses both hardened functions |
| All other modules | ✅ No impact: existing logic remains compatible |

## Behavior After Fix

Policies without a `modifiedDate` will now appear in the DCI report with:
- **Update date**: `NaT` → formatted as the raw timestamp conversion result
- **Time since last update**: `N/A`
- **Status**: counted as **not OK** (excluded from the OK policy count)
